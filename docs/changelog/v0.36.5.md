# v0.36.5 — 2026-05-10

**Make semantic runtime failure boundaries explicit and honest**

The semantic-retrieval restack had already introduced the local model
boundary, but several surrounding callers still treated corrupt semantic
config or sidecar state as if it were ordinary absence. This slice makes
those failure paths explicit: broken config imports and unreadable
embeddings sidecars now raise typed errors with path-rich context, while
only genuinely missing sidecars and missing NumPy continue to degrade
cleanly.

- Add `SemanticEmbeddingsLoadError` to `_semantic.runtime` and raise it when persisted semantic metadata or `.npy` sidecars are present but unreadable, while still returning ordinary absence when NumPy is unavailable or sidecars are simply missing.
- Route those typed failures deliberately at the owning boundaries: `search_index.py`, `evaluate_search.py`, and `construct_benchmark_fixture.py` surface semantic-unavailable errors, while the MCP-side `_load_embeddings(...)` helper logs the corruption, marks embeddings dirty, and lets the next refresh rebuild them.
- Tighten the semantic config boundary so CLI entry points use `load_config_checked(...)`, turning corrupt local semantic config into an explicit user-facing failure instead of silently treating it as disabled.
- Keep the build path honest by naming the optional array gate `_HAS_NUMPY`, so `build_index.py` states exactly which dependency can still degrade gracefully.
