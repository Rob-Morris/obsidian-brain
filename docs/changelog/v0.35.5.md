# v0.35.5 — 2026-05-04

**Maintain BM25 corpus stats incrementally per index update**

`_apply_pending_index_updates` previously called `_recompute_corpus_stats` after draining each batch, which rebuilt the document-frequency map by iterating every doc's term-frequency dict. Measured at ~29ms per mutation on a 948-doc, 24,530-term vault. The fix maintains `df`, `total_docs`, and `avg_dl` incrementally as docs are upserted, dropping per-mutation cost to ~0.3ms (96× speedup) while preserving exact equivalence with the full-recompute output.

- Add `_apply_doc_to_corpus_stats(index, doc, delta)` in `src/brain-core/scripts/build_index.py`; call it from `index_update` for both replace (-1/+1) and append (+1).
- Drop the unused `recompute` parameter from `index_update` and remove the trailing `_recompute_corpus_stats` call from `_apply_pending_index_updates` in `src/brain-core/brain_mcp/server.py`.
- Switch `_action_fix_links` single-file path to `mark_index_pending(path)` (vault-wide path still marks dirty).
- Add drift-assertion coverage in `tests/test_build_index.py` confirming incremental and full-recompute paths produce identical `corpus_stats`, plus a zero-`df` removal test and a 30-update churn test.
- Update DD-035 to reflect incremental corpus-stat maintenance and the single-file `fix-links` upsert path.
