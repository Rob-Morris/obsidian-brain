# v0.36.4 — 2026-05-10

**Make semantic model provisioning explicit and keep ordinary runtime paths fully local**

The earlier semantic-retrieval slices could enable the runtime and repair
sidecars, but they still treated the embedding model itself as an implicit
dependency discovered at query time. This release introduces `_semantic.model`
as the owned boundary for the pinned
`sentence-transformers/all-MiniLM-L6-v2@c9745ed1d9f207416be6d2e6f8de32d1f16199bf`
snapshot, records a local manifest under
`.brain/local/semantic-model-manifest.json`, and updates configure, repair,
check, build, search, process, and benchmark flows so they all load the same
vault-local model without surprise network fetches.

- Add the explicit model boundary: `_semantic.model` now owns the shipped model pin, vault-local snapshot path, manifest persistence, local-only loading, and cached query encoder lifecycle, while `_semantic.runtime` narrows back down to query-time helpers and sidecar inspection.
- Extend semantic provisioning and repair so the local `.venv`, pinned model snapshot, runtime marker, and embeddings sidecars converge as one lifecycle; `check.py` and `repair.py semantic` now inspect and report runtime drift, model drift, and sidecar drift separately.
- Update all semantic callers to the explicit `get_query_encoder(vault_root)` and `encode_query(vault_root, ...)` contract, including `search_index.py`, `process.py`, `build_index.py`, `evaluate_search.py`, and `construct_benchmark_fixture.py`, so every query path resolves against the provisioned local snapshot.
- Keep the ordinary runtime path honest from the start: `build_index.py` now treats `_HAS_NUMPY` as the only optional array gate, writes `model_revision` into `embeddings-meta.json`, and the user-facing docs consistently describe semantic provisioning as runtime install + local model snapshot + manifest + sidecars.
