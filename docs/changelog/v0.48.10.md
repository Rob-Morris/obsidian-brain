# v0.48.10

**Add non-MCP brain session resolution via a machine-level runtime**

**Machine-level resolution runtime**
- Add `_machine/resolution_runtime.py` and `_machine/resolve_brain.py`: a machine-owned, stdlib-only resolver that `brain session` runs before any Brain is selected, so an external workspace with a valid binding can bootstrap a session even when MCP is unavailable. `install.py` and `upgrade.py` provision and version-stamp the runtime under `~/.brain/resolution-runtime/` from the Brain being installed or upgraded, validating the full source closure before copying and keeping it in sync with the authored source.
- The resolver yields the workspace's bound Brain (or the machine default for an unbound workspace) or a structured error. It never executes another Brain's code, runtime, or resolver, even transiently, and never scans the registry for an arbitrary `session.py` host. See `docs/architecture/decisions/dd-054-machine-resolution-runtime.md`.

**Non-MCP session dispatch**
- `cli/brain` now resolves `brain session` through the machine runtime when no vault is directly in scope, then dispatches only to the resolved Brain's own `session.py` with an absolute `--vault`. Explicit `--vault` still dispatches directly; `BRAIN_WORKSPACE_DIR`, `--workspace-dir`, and the deprecated `--project-dir` anchor resolution; a stale `BRAIN_VAULT_ROOT` fails without falling back to another Brain.
- `brain session --json` emits a consistent degraded `session_resolution` payload — `vault_root: null`, the real resolver code, caller-observable context, and recovery guidance — across resolver and pre-resolver failures, including when no Python launcher is available. Remote Brain targets return explicit "not yet supported" guidance, preserving the seam for future remote session transport.
- Reframe `init.py` as legacy transport compatibility and update the project workspace bootstrap text to prefer MCP `brain_session`, with `brain session --json` as the offline fallback.

**Decisions, docs, and coverage**
