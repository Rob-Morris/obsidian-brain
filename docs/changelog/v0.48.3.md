# v0.48.3

**Preserve managed runtime launch paths across symlinks**

- `_common/_venv.py` now keeps the managed virtualenv launch path distinct from the native interpreter when deciding whether a precomputed launcher probe can be reused. Linux venvs often expose `bin/python` as a symlink to the system interpreter; Brain now probes the managed venv path instead of reusing native-Python dependency state and falsely reporting missing `mcp`.
- `_machine/topology.py` now uses the same executable launch-path rule when classifying exact versus compatible central runtimes, so different managed venv paths that symlink to the same interpreter binary are not collapsed into one exact runtime.
- Managed lifecycle result envelopes from `repair.py`, `configure.py`, `setup.py`, and semantic repair now report `sys.executable` as launched instead of `realpath(sys.executable)`, preserving the managed venv path in user-facing diagnostics.
- `_common.same_executable_path()` is now the single shared executable-identity predicate for managed-runtime launch-path checks. Symlink collapse remains intentional for unrelated filesystem canonicalisation cases such as path-security bounds, vault identity, process-family grouping, atomic-write targets, migration file de-duplication, CLI binary directories, and Python version-tag derivation.
- Tests cover dependency probing and machine-runtime classification across explicit venv-to-native-Python symlink boundaries (`tests/test_venv.py`, `tests/test_machine.py`).
- Move `BRAIN_INSTALL_REF` to `v0.48.3`; `BRAIN_CLI_VERSION` is unchanged because the CLI dispatch surface did not change.
