# v0.36.3 — 2026-05-09

**Round off semantic retrieval for installed vaults with a real configure/check/repair lifecycle**

The early semantic-retrieval slices made search and benchmarking work, but
installed vaults still lacked a finished lifecycle for local opt-in, health
checks, repair, and post-upgrade convergence. This release makes that lifecycle
explicit without changing the later model-boundary or warmup work that still
belongs in subsequent slices.

- Add the installed-vault semantic lifecycle surface: `configure.py semantic --enable`, `_lifecycle_common.py`, `_semantic/config.py`, and the `semantic` repair scope now provide a first-class path to enable, inspect, and repair semantic retrieval locally.
- Wire `install.sh --enable-semantic` through the same lifecycle so first-time installs can opt the vault in and provision the local semantic runtime in one step, with end-to-end install regression coverage.
- Keep runtime/config ownership explicit from the start: `_semantic.config` owns the local semantic flags and provisioning marker, while `_semantic.runtime` continues to expose the existing caller-facing surface until later slices move more modules over to the split boundary.
- Extend `upgrade.py` so semantic-intent vaults rerun `repair.py semantic --json` after the normal file update, keeping the local runtime and sidecars converged intentionally instead of drifting until the next query.
- Update the user-facing lifecycle docs so `configure.py semantic --enable`, `check.py --actionable`, and `repair.py semantic` read as the normal installed-vault workflow rather than as detached recovery details.
