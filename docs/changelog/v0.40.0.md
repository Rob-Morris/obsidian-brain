# v0.40.0 — 2026-05-14

**Rename repair.py index to lexical and let upgrade.py own repair**

This minor release tightens the lifecycle surface around retrieval repair and upgrade follow-up work. The retrieval-cache repair scope is now named for what it actually owns, and `upgrade.py` now runs the supported repair lifecycle itself instead of printing a raw `build_index.py` command and leaving the runtime choice to the caller.

- **Rename the retrieval-cache repair scope to `lexical`** — `src/brain-core/scripts/repair.py`, `src/brain-core/scripts/_repair_common.py`, `src/brain-core/scripts/_repair_runtime.py`, and `src/brain-core/scripts/_lifecycle_common.py` now use `lexical` as the canonical scope name for `.brain/local/retrieval-index.json`. Legacy `repair.py index` calls now fail with an explicit rename hint, and `tests/test_repair.py` locks that guidance in.
- **Let `upgrade.py` own post-upgrade search-asset repair** — `src/brain-core/scripts/upgrade.py` now runs `repair.py lexical` for lexical-only vaults or `repair.py semantic` for semantic-intent vaults after copy / compile / migration / definition-sync / runtime-sync work. The old raw `build_index.py` follow-up is gone; failures now surface the exact retry command for the chosen repair scope, unreadable post-upgrade semantic config resolves to a structured repair error instead of crashing the whole upgrade, compiled-router snapshot-seeding failures are logged as warnings, and `main()` now always writes one final `last-upgrade.json` seal after the optional CLI refresh step. `tests/test_upgrade.py` covers lexical and semantic upgrade paths, the `dependency_sync` → `search_asset_repair` stage sequence, the structured scope-detection error, and the no-CLI final-log path.
- **Keep the bundled installer wrapper aligned with the shipped release** — `cli/brain` and `README.md` now pin `brain install` and the public version badge to `v0.40.0`, while `tests/test_brain_cli.py` now derives the expected installer ref from `src/brain-core/VERSION` so future release bumps do not need a second hardcoded test edit.
