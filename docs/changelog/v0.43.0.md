# v0.43.0 — Unify artefact closure status vocabulary onto deprecated/parked

- Collapse type-specific abandonment statuses (`superseded`, `rejected`, `cancelled`, `blocked`) onto a single `deprecated` terminal across `living/design`, `temporal/plan`, `living/release`, and `living/task` (the types that actually carried granular abandonment vocab); the *reason* for closure now lives in a `> [!info] Deprecated — <reason>` callout in the artefact body rather than a granular status name.
- Introduce `deprecated` as an optional terminal across the other lifecycle types (`living/idea`, `temporal/idea-log`, `living/workspace`, `living/writing`, `living/journal`, `living/person`) so authors have a uniform closure path when one is needed.
- Normalise `parked` as the non-terminal pause state wherever pausing is meaningful; `living/task`'s `blocked` is renamed to `parked` (existing blocker context stays in the body).
- Update `src/brain-core/artefact-library/{living,temporal}/*/schema.yaml` and the matching `taxonomy.md` lifecycle tables, terminal-status sections, and frontmatter hints; refresh the catalogue lines in `src/brain-core/artefact-library/README.md`, `src/brain-core/guide.md`, and `src/brain-core/standards/extending/adding-temporal-type.md`; propagate the same content into `template-vault/_Config/Taxonomy/` via `make sync-template`.
- Add `src/brain-core/scripts/migrations/migrate_to_0_43_0.py` to rewrite retired statuses on the next vault upgrade, prepend or rewrite deprecation callouts (preserving any existing supersession links), collapse `+Superseded/`, `+Rejected/`, and `+Cancelled/` folders into `+Deprecated/`, and remove the retired folders once empty; the migration is idempotent.
- Add `tests/test_migrate_to_0_43_0.py` covering each status rewrite, folder collapse, callout preservation, idempotency, and dry-run; update `tests/test_compile_router.py::TestTemplateVault` assertions for the new release enum.
- Refresh `docs/user/system-guide.md` and `docs/user/workflows.md` so user-facing status references match the unified closure model; rationale and alternatives recorded in `docs/architecture/decisions/dd-050-closure-status-model.md`.
