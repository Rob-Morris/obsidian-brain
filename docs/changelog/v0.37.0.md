# v0.37.0 — 2026-05-11

**Relocate Python managed runtime to ~/.brain/venvs**

Empirical investigation of an iCloud-hosted vault found that ~11% of MCP cold starts (85 of 765 sessions) were SIGKILLed by Claude after its 30-second `initialize` timeout, with the slowest successful initialise sitting at exactly 29.0 s — at the cliff. The slow path is Python module imports against thousands of `site-packages/` files in `<vault>/.venv/`, materialised on demand from iCloud each cold start. The fix moves the managed runtime out of the vault.

- Adopted [DD-048](../architecture/decisions/dd-048-central-managed-runtime.md): the canonical managed runtime now lives at `~/.brain/venvs/py<X.Y>-<sha256(requirements.txt)[:16]>/`, machine-local and shared by any vault whose requirements hash matches. The launcher-vs-managed-runtime split from DD-043 is preserved; only the *location* of the managed runtime changes.
- New helper module `_common/_venv.py` is the single source of truth for the path rule. It exposes a Python API (`resolve_vault_venv_dir`, `ensure_central_venv`, etc.) and a CLI (`python _venv.py python|ensure --vault X --launcher Y`) so the bash installer never duplicates the rule.
- `install.sh` calls `_venv.py ensure` instead of running `python -m venv` directly. `upgrade.py`'s `_ensure_central_runtime` provisions the central venv when `requirements.txt` changes (creating or reusing) and surfaces a one-line migration command if the vault still has a legacy `.venv/`. `init.py`'s `find_python` prefers the central venv, falling back to the legacy vault `.venv/` so existing installs keep working until their owners run `init.py` to repoint MCP config. `repair.py` uses the helper to bootstrap the central runtime; its stdlib-only scopes pass `install_requirements=False` to skip the pip step (the same conditionality the previous code expressed).
- DD-043 header updated to "managed-runtime location superseded by DD-048 in v0.37.0"; DD index and Topic Map updated; `~/.brain/venvs/` is documented in `docs/architecture/overview.md`, `docs/architecture/security.md`, `docs/functional/scripts.md`, `docs/user/getting-started.md`, `docs/user/user-reference.md`, `src/brain-core/scripts/README.md`, and the contributor agent brief.
- Tests: new `tests/test_venv.py` covers the path rule, hash addressing, idempotent venv creation, the `install_requirements=False` skip, the `BRAIN_VENV_LAUNCHER` test seam, and the CLI subcommands. The repair-vault fixture now includes the helper. The install-script and upgrade-CLI integration tests use the seam to substitute a stub launcher and assert the central venv lands under a redirected `HOME`.

The vault-level decision is captured in [[Brain Setup And Bootstrapping Methodology]] D2 (vault) with the empirical backing in [[20260508-report~MCP Cold-Start Investigation iCloud Materialisation Of Vault-Local Venv]].

Existing installs keep their `<vault>/.venv/` until they run `init.py` against the new helper, which repoints MCP config at the central venv. The legacy directory may then be deleted; nothing in v0.37.0 references it. Cleanup of unreferenced central venvs is a deferred follow-up.
