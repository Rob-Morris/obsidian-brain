# v0.36.6 — 2026-05-10

**Detect router and embeddings drift across standalone compile paths**

The semantic runtime could now distinguish missing sidecars from corrupt
ones, but standalone compile and build flows still had no way to tell
whether persisted embeddings belonged to the current compiled router. This
slice adds an explicit router-source fingerprint to embeddings metadata and
teaches the standalone compile, MCP, and test surfaces to treat stale
router/embeddings pairings as drift that needs a rebuild.

- Add `ROUTER_SOURCE_HASH_KEY`, `router_source_hash(...)`, and `embeddings_meta_matches_router(...)` to `_semantic.runtime` so the router/source fingerprint rule lives alongside the rest of the embeddings-sidecar contract.
- Persist the compiled router's `meta.source_hash` into `embeddings-meta.json` during embedding builds so standalone `build_index.py` runs produce the same drift signal that later consumers can verify.
- Stop `compile_router.py` from eagerly deleting embeddings sidecars on every router rebuild; sidecars now remain on disk until an owning boundary compares the stored router fingerprint and decides whether a rebuild is needed.
- Teach the standalone semantic loader and the MCP owner boundary to reject stale router/embeddings pairings, surface them as unavailable or dirty as appropriate, and rebuild them on the next refresh, with regression coverage for the stale-sidecar path.
