# v0.35.8 — 2026-05-07

**Fix double-date filenames from migrate_naming catchall path**

`migrate_naming.compute_new_filename` recognises legacy filename forms via three regexes (`_OLD_TEMPORAL_RE`, `_OLD_PREFIXLESS_RE`, `_OLD_LIVING_RE`) — all three require dash-separated, lowercase shapes. When a filename has the right *content* but wrong *separator* (e.g. `20260322 Canary Md Article.md` — space-separated leading date with title-case title), every legacy regex fails to match and execution falls through to a catchall that uses `os.path.splitext(filename)[0]` as the title. `render_filename` then re-applies the canonical `YYYYMMDD-` prefix from the artefact's `created` date on top of a title that already contains a date, producing names like `20260322-20260322 Canary Md Article.md`. Two real Writing/+Published files were renamed this way during a force-upgrade sync.

- Add `_LEADING_DATE_IN_TITLE_RE` (`^(\d{8})[ \-]+`) and apply it in the catchall: when the stem starts with an 8-digit date followed by space or dash, strip the prefix from the title and seed `historical_date` if no `created` is in frontmatter. The naming engine then re-renders the canonical form without doubling the prefix.
- Add `test_space_separated_date_prefix_not_double_prefixed` covering the exact regression — a space-separated date prefix renders as `20260322-Canary Md Article.md`, not `20260322-20260322 Canary Md Article.md`.
- Add `test_dash_separated_date_prefix_with_capitalised_title` covering the no-op path — already-canonical `20260322-Canary Md Article.md` returns `None` (no migration) because `validate_filename` accepts it.

The fix is forward-only — the migration now declines to produce double-date filenames, but already-double-prefixed filenames in vaults from earlier runs need to be repaired manually or by reverting the rename and re-running migration with this fix.
